# Context Ring & TSCC Popover Bugfix Design

## Overview

Two bugs prevent live session telemetry from reaching the user. Bug 1: the backend context warning bridge in `session_unit.py` filters out `context_warning` SSE events at the "ok" level, so the frontend never receives usage data when context is below 70%, leaving the ContextUsageRing permanently gray. Bug 2: the TSCC Popover Button shows empty/disabled content because the `useTSCCState` hook only fetches system prompt metadata via a separate API call on `sessionId` change — which fires before the backend has populated metadata during streaming — and never receives updated metadata after the conversation turn completes.

The fix strategy uses a unified SSE data delivery mechanism for both features: (1) remove the `!= "ok"` filter in the backend so `context_warning` events at all levels are emitted, (2) emit a new `system_prompt_metadata` SSE event from the backend alongside the `context_warning` event after the result message, and (3) handle both events in the frontend SSE handler (`useChatStreamingLifecycle.ts`), storing them on `tabMapRef`. This eliminates the separate API fetch in `useTSCCState` and the timing/race condition entirely — both features get their data from the same SSE pipeline.

## Glossary

- **Bug_Condition (C)**: Two conditions — C1: a `context_warning` event with `level == "ok"` is generated but filtered out before emission; C2: system prompt metadata is fetched via separate API call that races with backend population, and is never refreshed after streaming completes
- **Property (P)**: P1: all non-None `context_warning` events are emitted regardless of level; P2: system prompt metadata is delivered via SSE `system_prompt_metadata` event after each conversation turn, stored on `tabMapRef`, and read by the TSCC popover from tab state
- **Preservation**: Warn/critical events continue to be emitted; tab isolation and popover auto-close behavior unchanged; `GET /api/chat/{session_id}/system-prompt` endpoint remains available (not removed)
- **`build_context_warning()`**: Classmethod in `backend/core/prompt_builder.py` that computes context usage percentage and returns an SSE event dict with level ok/warn/critical
- **Context warning bridge**: Code block in `session_unit.py` `_run_query_on_client()` that reads `input_tokens` from the result event and yields a `context_warning` SSE event
- **`system_prompt_metadata` SSE event**: New SSE event emitted by the backend alongside `context_warning`, containing the assembled system prompt metadata for the session
- **`useChatStreamingLifecycle`**: Hook in `desktop/src/hooks/useChatStreamingLifecycle.ts` that handles SSE events and manages streaming state per tab
- **`useTSCCState`**: React hook in `desktop/src/hooks/useTSCCState.ts` that manages TSCC lifecycle state and expand/pin preferences (simplified — no longer fetches metadata)
- **Display mirror pattern**: `tabMapRef` is the authoritative store; React `useState` is a display mirror updated only for the active tab

## Bug Details

### Bug Condition

Two independent bug conditions exist:

**Bug 1 — Context Warning Filter**: The context warning bridge in `session_unit.py` line 982 filters out events where `level == "ok"`, preventing the frontend from receiving context usage data when usage is below 70%.

**Bug 2 — TSCC Metadata Delivery**: The `useTSCCState` hook fetches metadata via a separate `GET /api/chat/{session_id}/system-prompt` API call only on `sessionId` change, which fires before the backend has populated metadata during the streaming loop. No refresh occurs after streaming completes. This is a fundamentally different data delivery mechanism than the context ring (which uses SSE), creating a timing/race condition.

**Formal Specification:**
```
FUNCTION isBugCondition_ContextWarning(input)
  INPUT: input of type {input_tokens: int, model_name: str}
  OUTPUT: boolean

  warning_evt := PromptBuilder.build_context_warning(input.input_tokens, input.model_name)
  RETURN warning_evt IS NOT None
         AND warning_evt.level == "ok"
         AND event_is_not_yielded(warning_evt)
END FUNCTION

FUNCTION isBugCondition_TSCCMetadata(input)
  INPUT: input of type {sessionId: str, streamingCompleted: bool}
  OUTPUT: boolean

  RETURN input.sessionId IS NOT None
         AND input.streamingCompleted == true
         AND metadata_not_delivered_via_SSE(input.sessionId)
END FUNCTION
```

### Examples

- **Ok-level filtered**: `input_tokens=5000`, model with 1M window → `pct=0` (rounds to 0), `level="ok"` → event generated by `build_context_warning()` but NOT yielded by the bridge → ContextUsageRing stays gray
- **Low usage with 1M window**: `input_tokens=5000`, model `claude-opus-4-6` with 1M window → `pct=1`, `level="ok"` → ring shows green with barely any fill arc (correct behavior once fix is applied)
- **Warn-level passes (not a bug)**: `input_tokens=150000`, model with 200K window → `pct=75`, `level="warn"` → event IS yielded → ring shows yellow (this path works correctly)
- **Metadata fetch too early**: New session starts → `session_start` SSE arrives → `sessionId` changes → `useTSCCState` fetches metadata via API → backend hasn't populated it yet → returns empty/default
- **No metadata refresh after streaming**: Streaming completes → `result` SSE arrives → metadata now exists on backend → `useTSCCState` never re-fetches → popover shows empty content

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Warn-level (≥70%) and critical-level (≥90%) `context_warning` SSE events must continue to be emitted with the same level, message, and percentage values
- Frontend display of warn/critical context warnings (banner/toast + ring color) must remain unchanged
- `build_context_warning()` must continue to return `None` for null, zero, or negative `input_tokens`
- Tab switching must continue to restore `contextWarning` state from `tabMapRef` (display mirror pattern)
- TSCC popover auto-close on tab switch via `sessionId` change detection must remain unchanged
- `GET /api/chat/{session_id}/system-prompt` endpoint remains available (not removed) for backward compatibility

**Scope:**
All inputs that do NOT involve the two bug conditions should be completely unaffected by this fix. This includes:
- All non-context-warning SSE events (assistant, tool_use, tool_result, result, error, etc.)
- Mouse/keyboard interactions with the chat interface
- Tab switching, session creation, session resumption flows
- All other hooks and components not mentioned in the fix

## Hypothesized Root Cause

Based on the bug description and code analysis, the root causes are:

1. **Overly Aggressive Filter in Context Warning Bridge**: In `session_unit.py` line 982, the condition `if warning_evt and warning_evt.get("level") != "ok"` was likely added intentionally to suppress "ok" events as noise, but the frontend needs all levels to display the context usage ring. The `build_context_warning()` function in `prompt_builder.py` already correctly generates events at all levels including "ok" — the bridge just drops them.

2. **Different Data Delivery Mechanisms for Related Features**: The context ring gets data via SSE (`context_warning` events), but the TSCC popover gets data via a separate REST API call (`GET /api/chat/{session_id}/system-prompt`). This creates a timing mismatch — the API call fires on `sessionId` change (before metadata is populated), while SSE events arrive after the streaming loop completes (when metadata IS available). Using the same SSE pipeline for both eliminates this class of bug entirely.

3. **Race Condition on Initial API Fetch**: The `sessionId` is set when `session_start` SSE arrives, but `_system_prompt_metadata` is populated by `session_router.py` during `build_options()` — which happens before streaming starts. However, the metadata may not be fully populated until the prompt is assembled. The initial API fetch races with this assembly. With SSE delivery, the metadata is emitted at the end of the streaming loop when it's guaranteed to be available.

## Correctness Properties

Property 1: Bug Condition - All Valid Context Warning Events Are Emitted

_For any_ input where `build_context_warning(input_tokens, model_name)` returns a non-None event (i.e., `input_tokens > 0` and a valid model), the context warning bridge in `session_unit.py` SHALL yield that event regardless of its `level` value (ok, warn, or critical).

**Validates: Requirements 2.1, 2.2**

Property 2: Bug Condition - System Prompt Metadata Delivered via SSE

_For any_ session where a conversation turn completes (result event is emitted) and `session_registry.system_prompt_metadata[session_id]` contains metadata, the backend SHALL yield a `system_prompt_metadata` SSE event containing that metadata, and the frontend SSE handler SHALL store it on `tabMapRef.promptMetadata` and expose it as a display mirror for the TSCC popover.

**Validates: Requirements 2.3, 2.4**

Property 3: Preservation - Warn and Critical Events Unchanged

_For any_ input where `build_context_warning(input_tokens, model_name)` returns an event with `level` equal to "warn" or "critical", the fixed context warning bridge SHALL yield that event with identical content to the original (unfixed) bridge, preserving all existing warn/critical behavior.

**Validates: Requirements 3.1, 3.2**

Property 4: Preservation - No Event for Invalid Input Tokens

_For any_ input where `input_tokens` is null, zero, or negative, `build_context_warning()` SHALL continue to return `None` and no `context_warning` SSE event shall be emitted.

**Validates: Requirements 3.3**

Property 5: Preservation - Tab Isolation for Metadata

_For any_ tab switch, the `promptMetadata` stored on `tabMapRef` SHALL be correctly isolated per tab, and the display mirror SHALL update to show the active tab's metadata (same pattern as `contextWarning`).

**Validates: Requirements 3.4, 3.6**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct:

### Backend: Emit both SSE events after result message

**File**: `backend/core/session_unit.py`

**Function**: `_run_query_on_client` (context warning bridge block, ~line 970-985)

**Specific Changes**:
1. **Remove the "ok" level filter**: Change the condition from `if warning_evt and warning_evt.get("level") != "ok":` to simply `if warning_evt:`. This allows events at all levels (ok, warn, critical) to be yielded to the SSE stream.
2. **Update the comment**: The existing comment says "if above 70% of the model's context window" — update to reflect that events are emitted at all usage levels.
3. **Emit `system_prompt_metadata` SSE event**: After yielding the `context_warning` event, also yield a `system_prompt_metadata` event containing the metadata from `session_registry.system_prompt_metadata[self.session_id]`.

**Before:**
```python
# ── Context usage warning bridge ───────────────────
# Read input_tokens from the result event and emit a
# context_warning SSE event if above 70% of the model's
# context window.  This wires the dead-code warning
# builder into the live streaming path.
input_tokens = (usage.get("input_tokens") if usage else None)
if input_tokens and input_tokens > 0:
    try:
        from .prompt_builder import PromptBuilder
        warning_evt = PromptBuilder.build_context_warning(
            input_tokens, self._model_name
        )
        if warning_evt and warning_evt.get("level") != "ok":
            yield warning_evt
    except Exception:
        pass  # Never block on warning failure
```

**After:**
```python
# ── Context usage & metadata bridge ────────────────
# Read input_tokens from the result event and emit a
# context_warning SSE event at all usage levels (ok,
# warn, critical).  Also emit system_prompt_metadata
# so the frontend TSCC popover gets data via the same
# SSE pipeline — no separate API fetch needed.
input_tokens = (usage.get("input_tokens") if usage else None)
if input_tokens and input_tokens > 0:
    try:
        from .prompt_builder import PromptBuilder
        warning_evt = PromptBuilder.build_context_warning(
            input_tokens, self._model_name
        )
        if warning_evt:
            yield warning_evt
    except Exception:
        pass  # Never block on warning failure

# Emit system prompt metadata for TSCC popover
try:
    from . import session_registry
    spm = session_registry.system_prompt_metadata.get(
        self.session_id
    )
    if spm:
        yield {"type": "system_prompt_metadata", **spm}
except Exception:
    pass  # Never block on metadata failure
```

### Frontend: Handle both SSE events in streaming lifecycle

**File**: `desktop/src/hooks/useChatStreamingLifecycle.ts`

**Specific Changes**:
1. **Handle `system_prompt_metadata` SSE events**: Add a handler alongside the existing `context_warning` handler. Store the metadata on `tabMapRef` as `promptMetadata`. Mirror to React state only for the active tab (same display mirror pattern as `contextWarning`).
2. **Expose `promptMetadata` as a display mirror**: Add `promptMetadata` state and expose it from the hook return value, alongside `contextWarning`.

**New SSE handler (added after the `context_warning` handler):**
```typescript
// System prompt metadata — backend emits after each turn alongside
// context_warning.  Same display mirror pattern: write to tabMapRef,
// mirror to React state only for the active tab.
else if (event.type === 'system_prompt_metadata') {
  const { type, ...metadata } = event;
  const tabState = capturedTabId
    ? tabMapRef.current.get(capturedTabId)
    : undefined;
  if (tabState) {
    tabState.promptMetadata = metadata as SystemPromptMetadata;
  }
  if (capturedTabId === null || capturedTabId === activeTabIdRef.current) {
    setPromptMetadata(metadata as SystemPromptMetadata);
  }
}
```

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bugs on unfixed code, then verify the fixes work correctly and preserve existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate both bugs BEFORE implementing the fix. Confirm or refute the root cause analysis. If we refute, we will need to re-hypothesize.

**Test Plan**: Write tests that exercise the context warning bridge with ok-level inputs and verify events are dropped. Write tests that verify no `system_prompt_metadata` SSE event is emitted by the current code. Run these tests on the UNFIXED code to observe failures and understand the root cause.

**Test Cases**:
1. **Ok-Level Filter Test (Backend)**: Call `_run_query_on_client` with `input_tokens` that produce an ok-level warning → verify the event is NOT yielded (will confirm the bug on unfixed code)
2. **Warn-Level Passthrough Test (Backend)**: Call with `input_tokens` that produce a warn-level warning → verify the event IS yielded (confirms warn path works)
3. **No Metadata SSE Test (Backend)**: Verify that the current streaming path does NOT yield a `system_prompt_metadata` event (will confirm the bug — metadata only available via separate API)
4. **Metadata API Timing Test (Frontend)**: Verify that `useTSCCState` fetches metadata on `sessionId` change but gets empty/default because backend hasn't populated it yet

**Expected Counterexamples**:
- Ok-level context_warning events are generated by `build_context_warning()` but silently dropped by the `!= "ok"` filter
- No `system_prompt_metadata` SSE event exists in the streaming path — metadata is only available via separate API call that races with backend population

### Fix Checking

**Goal**: Verify that for all inputs where the bug conditions hold, the fixed functions produce the expected behavior.

**Pseudocode:**
```
-- Bug 1: Context Warning
FOR ALL (input_tokens, model_name) WHERE build_context_warning(input_tokens, model_name) IS NOT None DO
  events := collect(fixed_streaming_path(input_tokens, model_name))
  ASSERT context_warning_event IN events
  ASSERT events.context_warning.level IN ["ok", "warn", "critical"]
  ASSERT events.context_warning.pct == expected_pct(input_tokens, model_name)
END FOR

-- Bug 2: System Prompt Metadata via SSE
FOR ALL session WHERE session_registry.system_prompt_metadata[session.id] IS NOT None DO
  events := collect(fixed_streaming_path(session))
  ASSERT system_prompt_metadata_event IN events
  ASSERT events.system_prompt_metadata == session_registry.system_prompt_metadata[session.id]
END FOR
```

```
FUNCTION expectedBehavior(result)
  -- For Bug 1:
  INPUT: result of type {events: List[SSEEvent], input_tokens: int, model_name: str}
  OUTPUT: boolean

  expected_evt := build_context_warning(result.input_tokens, result.model_name)
  IF expected_evt IS NOT None THEN
    RETURN context_warning_event_in(result.events)
           AND result.events.context_warning == expected_evt
  ELSE
    RETURN context_warning_event_not_in(result.events)
  END IF

  -- For Bug 2:
  INPUT: result of type {events: List[SSEEvent], session_id: str}
  OUTPUT: boolean

  expected_meta := session_registry.system_prompt_metadata.get(result.session_id)
  IF expected_meta IS NOT None THEN
    RETURN system_prompt_metadata_event_in(result.events)
           AND result.events.system_prompt_metadata == expected_meta
  ELSE
    RETURN true  -- no metadata to emit is fine
  END IF
END FUNCTION
```

### Preservation Checking

**Goal**: Verify that for all inputs where the bug conditions do NOT hold, the fixed functions produce the same result as the original functions.

**Pseudocode:**
```
-- Preservation: warn/critical events unchanged
FOR ALL (input_tokens, model_name) WHERE build_context_warning(input_tokens, model_name).level IN ["warn", "critical"] DO
  ASSERT fixed_bridge_yield(input_tokens, model_name) == original_bridge_yield(input_tokens, model_name)
END FOR

-- Preservation: invalid input_tokens still produce no event
FOR ALL input_tokens WHERE input_tokens IS None OR input_tokens <= 0 DO
  ASSERT fixed_bridge_yield(input_tokens, model_name) == []
  ASSERT original_bridge_yield(input_tokens, model_name) == []
END FOR

-- Preservation: tab isolation for metadata
FOR ALL (tabA, tabB) WHERE tabA != tabB DO
  ASSERT tabMapRef.get(tabA).promptMetadata != tabMapRef.get(tabB).promptMetadata
         OR both are from the same session
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many random `input_tokens` values across the full range (1 to 1M+) to verify all levels are handled correctly
- It catches edge cases at threshold boundaries (69.9% vs 70.0% vs 70.1%)
- It provides strong guarantees that warn/critical behavior is unchanged for all non-ok inputs

**Test Plan**: Observe behavior on UNFIXED code first for warn/critical events, then write property-based tests capturing that behavior.

**Test Cases**:
1. **Warn/Critical Event Preservation**: Generate random `input_tokens` values that produce warn/critical levels → verify events are yielded identically before and after fix
2. **Invalid Input Preservation**: Generate random null/zero/negative `input_tokens` → verify no events are emitted before or after fix
3. **Tab Isolation Preservation**: Verify `contextWarning` and `promptMetadata` state is correctly isolated per tab on `tabMapRef`
